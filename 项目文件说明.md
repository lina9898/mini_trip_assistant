# 二、当前项目总体定位

到现在为止，我们的智能旅行助手已经不是最初的命令行小程序了。

它现在更像是：

```text
一个基于 LLM + Tools + Memory + Storage + Export 的智能旅行规划 Agent 原型
```

核心能力包括：

```text
1. 采集用户旅行需求
2. 规划工具调用
3. 执行多个外部工具
4. 综合工具结果生成行程
5. 多轮修改行程
6. 记住用户持续约束
7. 恢复历史版本
8. 保存项目快照
9. 导出 Markdown / HTML / PDF
10. 检查图片、开放时间、酒店、交通、活动等真实旅行约束
```

所以后面做 Web，不是重新做一个项目，而是把这些能力封装成接口和页面。

# 四、每个文件负责什么？

## 1. `main.py`

只负责启动程序。

```python
from app import run_app

if __name__ == "__main__":
    run_app()
```

它不应该再写业务逻辑。

---

## 2. `app.py`

负责命令行应用主流程。

主要职责：

```text
显示首页菜单
创建新行程
读取已有行程
进入当前行程操作菜单
处理修改、保存、导出、恢复上一版、查看记忆等操作
```

它相当于 CLI 版的“控制器”。

后面做 FastAPI 时，`app.py` 的很多逻辑会被 API 路由取代。

---

## 3. `menu.py`

负责命令行输入。

主要职责：

```text
输入出发地
输入目的地
输入计划出行日期
输入旅行天数
输入人数
输入偏好
输入节奏
输入预算档位
显示首页菜单
显示当前行程操作菜单
```

后面做 Web 端时，`menu.py` 基本会被前端表单替代。

---

## 4. `trip_service.py`

这是当前最核心的业务服务层。

主要职责：

```text
创建项目 create_project()
读取项目 load_project_flow()
修改项目 edit_project()
保存项目 save_current_project()
导出项目 export_current_project()
打印项目 print_project()
调用生成前工具
调用生成后工具
组装 project_data
```

后面 FastAPI 接口化时，最重要的就是复用 `trip_service.py`。

比如：

```text
POST /api/trips/generate  → 调 create_project()
POST /api/trips/{id}/edit → 调 edit_project() 的非 input 版本
POST /api/trips/{id}/export → 调 export_current_project()
```

所以第 27 节会重点改造 `trip_service.py`，把里面依赖 `input()` 的部分抽出来。

---

## 5. `planner.py`

负责调用大模型生成最终旅行计划。

主要职责：

```text
接收用户输入
接收工具结果
调用 build_trip_plan_prompt()
调用 LLM
解析 JSON
返回 trip_data
```

它只管“生成完整行程”，不管保存、不管导出、不管命令行。

---

## 6. `prompts.py`

负责所有 Prompt 构造。

主要包括：

```text
工具规划 Prompt
行程生成 Prompt
行程修改 Prompt
```

现在里面已经包含很多规则：

```text
天气不能编造远期天气
必须遵守用户记忆
必须参考 POI
必须参考路线距离
酒店不能编造价格库存
交通不能编造车次航班
活动不能编造票价余票
```

这个文件很重要，后面 Web 化时仍然保留。

---

## 7. `llm_client.py`

负责封装大模型调用。

主要职责：

```text
读取 LLM_API_KEY
读取 LLM_BASE_URL
读取 LLM_MODEL
发送 messages
返回模型输出
```

这个文件应该保持稳定，不要掺杂业务逻辑。

---

## 8. `tool_planner.py`

负责让大模型规划应该调用哪些工具。

输入：

```text
origin
destination
start_date
days
preference
pace
budget_level
people
```

输出：

```json
{
  "tool_calls": [...]
}
```

它解决的是：

```text
这个旅行需求需要调用哪些工具？
```

---

## 9. `tool_executor.py`

负责执行工具计划。

主要职责：

```text
按顺序执行工具
解析 {{map_tool.adcode}}
解析 {{poi_tool}}
保存 tool_results
保存 execution_logs
```

它是 Agent 工具调用链的执行器。

---

## 10. `tool_plan_validator.py`

负责校验和补全工具计划。

主要职责：

```text
检查必要工具是否缺失
补全 map_tool / weather_tool / budget_tool / poi_tool / route_tool 等
修复参数
调整工具顺序
```

它是兜底机制，避免大模型漏掉关键工具。

---

## 11. `memory_manager.py`

负责项目级记忆。

主要职责：

```text
保存用户持续约束
保存避免地点
保存偏好记忆
保存行程版本历史
恢复上一版
查看当前记忆
从用户修改要求中提取记忆
```

它解决两个核心问题：

```text
1. 用户说“不去外滩”，后面不能忘
2. 用户说“恢复上一版”，系统能准确恢复
```

---

## 12. `storage.py`

负责项目保存、读取和快照管理。

主要职责：

```text
构造 project_data
保存项目快照
读取历史项目
列出历史项目
标记 dirty
确保导出前保存
记录导出历史
```

现在保存逻辑是：

```text
首次保存生成 v1
未修改多次导出仍覆盖 v1
修改后保存或导出生成 v2
v2 后继续导出仍覆盖 v2
再次修改后生成 v3
```

这是后面 Web 历史项目页的基础。

---

## 13. `exporter.py`

负责导出报告。

主要导出：

```text
Markdown
HTML
PDF
```

HTML / PDF 报告中现在可以展示：

```text
基础信息
出行日期
天气
预算
每日行程
地点图片
酒店建议
城际交通建议
开放时间检查
城市活动提醒
导出说明
```

---

## 14. `templates.py`

负责命令行格式化展示。

主要是把 `trip_data` 和部分工具结果格式化成可读文本。

后面做 Web 后，`templates.py` 作用会变小，但命令行版仍然需要它。

---

## 15. `date_utils.py`

负责日期处理。

主要职责：

```text
校验 YYYY-MM-DD
计算 end_date
生成 travel_dates
判断出行日期距离今天几天
```

它支撑第 21 节的天气逻辑。

# 五、tools 文件夹职责总览

## 1. `map_tool.py`

根据目的地查地图信息：

```text
经纬度
城市
adcode
格式化地址
```

主要给天气、酒店、交通等工具提供基础地理信息。

---

## 2. `weather_tool.py`

根据 `adcode + start_date + days` 查询天气。

模式包括：

```text
live
forecast
out_of_range
past_date
invalid_date
```

核心原则：

```text
不能把今天的天气当作未来旅行当天的天气
不能编造远期天气
```

---

## 3. `budget_tool.py`

根据：

```text
days
people
budget_level
```

估算：

```text
住宿
餐饮
交通
门票
单人总预算
总预算
```

---

## 4. `poi_tool.py`

根据目的地和偏好查询真实地点。

返回：

```text
景点
餐饮
商圈
公园
博物馆
POI 坐标
POI 类型
POI 图片
```

它让行程从“模型凭常识生成”变成“参考真实地点生成”。

---

## 5. `route_tool.py`

根据 POI 坐标计算地点之间距离。

作用：

```text
判断哪些地点适合放在同一天
判断哪些地点较远
辅助 route_strategy
```

---

## 6. `image_tool.py`

根据最终行程中的具体地点匹配图片。

现在不是城市级随机配图，而是：

```text
外滩 → 外滩图
武康路 → 武康路图
豫园 → 豫园图
```

优先使用：

```text
高德 POI photos
```

没有再用：

```text
Unsplash
```

---

## 7. `hotel_tool.py`

根据行程核心景点推荐住宿区域和酒店候选。

作用：

```text
搜索酒店 POI
计算酒店到核心景点距离
生成住宿区域建议
```

---

## 8. `transport_tool.py`

根据出发地和目的地生成城际交通建议。

作用：

```text
解析出发地和目的地坐标
估算驾车距离和耗时
尝试公共交通路线
推荐高铁 / 飞机 / 自驾 / 城际交通
影响第一天行程强度
```

---

## 9. `opening_hours_tool.py`

根据最终行程地点检查开放时间。

作用：

```text
查询 POI business.opentime_today
查询 opentime_week
判断是否闭馆
判断晚上是否适合去
输出 high / medium / low 风险
```

---

## 10. `event_tool.py`

根据目的地和出行日期查询城市活动。

作用：

```text
演唱会
展览
体育赛事
音乐节
漫展
城市节庆
```

目前主要作为可扩展工具，国内活动数据源可能还不够稳定，所以必须遵守：

```text
查不到就提示，不编造活动
```

# 六、完整运行流程整理

现在创建一个新行程的完整流程如下：

```text
用户输入旅行需求
   ↓
menu.py 收集 origin / destination / start_date / days / people / preference / pace / budget_level
   ↓
trip_service.create_project()
   ↓
date_utils 生成 travel_dates
   ↓
tool_planner 让 LLM 规划工具调用
   ↓
tool_plan_validator 校验和补全工具计划
   ↓
tool_executor 执行生成前工具
   ↓
获得 tool_results：
   map_info
   weather_info
   budget
   poi_info
   route_info
   hotel_info
   transport_info
   event_info
   ↓
planner.generate_trip_plan()
   ↓
LLM 综合用户需求 + 工具结果生成 trip_data
   ↓
生成后工具执行：
   opening_hours_tool 检查开放时间
   image_tool 匹配地点图片
   ↓
storage.build_project_data()
   ↓
memory_manager 初始化记忆和版本历史
   ↓
add_version 保存初始版本
   ↓
命令行展示行程
   ↓
用户选择保存 / 修改 / 导出 / 查看记忆 / 恢复上一版
```

# 七、生成前工具和生成后工具要区分清楚

这是第 26 节最重要的整理之一。

## 生成前工具

这些工具应该在大模型生成行程之前执行，因为它们会影响行程生成：

```text
map_tool
weather_tool
budget_tool
poi_tool
route_tool
hotel_tool
transport_tool
event_tool
```

它们的结果会传给 `generate_trip_plan()`。

---

## 生成后工具

这些工具更适合在行程生成之后执行，因为它们依赖最终 `trip_data.plan` 中的具体地点：

```text
image_tool
opening_hours_tool
```

原因是：

```text
图片匹配需要知道最终行程用了哪些地点
开放时间检查也需要知道最终行程安排了哪些地点和时段
```

所以它们不应该强制放进 `REQUIRED_BASE_TOOLS`。

# 十、当前还存在的工程问题

进入 FastAPI 前，需要注意几个问题。

## 1. `trip_service.py` 里还有 `input()`

比如：

```python
user_edit_request = input("请输入你的修改要求：").strip()
```

这在命令行里没问题，但 FastAPI 不能用 `input()`。

所以第 27 节要把它拆成两个函数：

```text
edit_project_cli()
edit_project_by_request(project_data, user_edit_request)
```

Web 接口调用后者。

---

## 2. `load_project_flow()` 依赖命令行选择

现在读取历史项目可能是：

```text
列出项目
让用户输入编号
读取对应项目
```

Web 端不能这样。

后面要拆成：

```text
list_projects()
load_project_by_filepath()
load_project_by_id()
```

---

## 3. 导出函数返回的是本地路径

Web 端需要的是：

```text
下载链接
文件路径
文件类型
导出状态
```

所以第 27 节要让导出接口返回 JSON：

```json
{
  "success": true,
  "export_type": "pdf",
  "filepath": "exports/xxx.pdf",
  "download_url": "/downloads/xxx.pdf"
}
```

---

## 4. 长时间运行问题

生成完整行程会调用很多工具和 LLM，可能比较慢。

Web 端要考虑：

```text
按钮 loading 状态
生成中提示
错误提示
超时处理
```

第 27 节先做同步接口，后面再考虑异步任务。

# 十二、第 26 节小结

第 26 节我们没有急着写 Web，而是先做了一次项目总整理。

这一节明确了：

```text
当前项目文件结构
每个文件的职责
完整运行流程
生成前工具和生成后工具
project_data 最终结构
前端未来需要哪些字段
进入 FastAPI 前需要拆掉哪些 input() 逻辑
```

系统现在已经进入：

```text
Agent 产品化前的工程整理阶段
```

第 26 节最重要的一句话是：

> 在进入 Web 化之前，必须先把项目结构、数据结构和运行流程整理清楚；否则前端页面只会把后端混乱放大，而不是让产品更清晰。

下一节就可以正式开始：

```text
第 27 节：FastAPI 后端接口化
```
