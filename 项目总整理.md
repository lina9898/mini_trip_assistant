# 第 26 节：项目总整理，为 Web 化做准备

## 一、当前项目定位

当前项目已经不只是命令行小程序，而是一个基于 `LLM + Tools + Memory + Storage + Export` 的智能旅行规划 Agent 原型。

它目前具备的核心能力包括：

- 采集用户旅行需求
- 规划工具调用
- 执行多个外部工具
- 综合工具结果生成行程
- 多轮修改行程
- 记住用户持续约束
- 恢复历史版本
- 保存项目快照
- 导出 Markdown / HTML / PDF
- 检查图片、开放时间、酒店、交通、活动等真实旅行约束

## 三、文件职责

### 入口与交互

- `main.py`
  只负责启动程序，不承载业务逻辑。

- `app.py`
  命令行主流程控制器，负责首页菜单、进入当前项目菜单、修改、保存、导出、恢复版本、查看记忆。

- `menu.py`
  负责 CLI 输入与菜单展示，后续 Web 化时会被表单替代。

### 业务服务层

- `trip_service.py`
  当前最核心的业务服务层，负责创建项目、修改项目、保存项目、导出项目、组装 `project_data`，以及串起生成前工具和生成后工具。

- `planner.py`
  负责调用大模型生成最终旅行计划。

- `itinerary_editor.py`
  负责调用大模型修改已有行程。

- `prompts.py`
  负责工具规划 Prompt、行程生成 Prompt、行程修改 Prompt。

- `llm_client.py`
  负责封装模型调用，只处理消息发送和模型返回。

### Agent 工具链

- `tool_planner.py`
  让 LLM 规划应该调用哪些工具。

- `tool_executor.py`
  按顺序执行工具计划，解析 `{{map_tool.adcode}}`、`{{poi_tool}}` 等占位符。

- `tool_plan_validator.py`
  校验、补全、排序工具计划，避免大模型漏掉关键工具。

### 项目状态与持久化

- `memory_manager.py`
  负责持续约束、避免地点、偏好记忆、版本历史、恢复上一版。

- `storage.py`
  负责 `project_data` 构造、项目快照保存、读取、列出历史项目、导出历史记录。

- `date_utils.py`
  负责日期校验、`end_date` 计算、`travel_dates` 生成、日期差计算。

### 展示与导出

- `templates.py`
  负责命令行文本格式化展示。

- `exporter.py`
  负责 Markdown / HTML / PDF 导出。


## 四、tools 职责总览

- `map_tool.py`
  查询经纬度、城市、`adcode`、格式化地址。

- `weather_tool.py`
  根据 `adcode + start_date + days` 查询 `live / forecast / out_of_range / invalid_date` 天气模式。

- `budget_tool.py`
  估算住宿、餐饮、交通、门票、人均和总预算。

- `poi_tool.py`
  返回真实景点、餐饮、商圈、公园、博物馆、POI 坐标与图片。

- `route_tool.py`
  计算地点间距离和通勤时间，辅助路线合理性。

- `image_tool.py`
  根据最终行程中的具体地点匹配图片，优先高德 POI 图片，再用 Unsplash 兜底。

- `hotel_tool.py`
  搜索酒店候选，计算到核心景点距离，生成住宿区域建议。

- `transport_tool.py`
  处理出发地到目的地的大交通建议，影响第一天强度安排。

- `opening_hours_tool.py`
  根据最终行程的具体地点与时段，检查开放时间风险。

- `event_tool.py`
  按目的地和日期范围查询城市活动，作为兴趣补充与拥堵风险提醒。


## 五、完整运行流程

```text
用户输入旅行需求
   ↓
menu.py 收集 origin / destination / start_date / days / people / preference / pace / budget_level
   ↓
trip_service.create_project()
   ↓
date_utils 生成 travel_dates
   ↓
tool_planner 让 LLM 规划工具调用
   ↓
tool_plan_validator 校验和补全工具计划
   ↓
tool_executor 执行生成前工具
   ↓
获得 tool_results：
   map_info
   weather_info
   budget
   poi_info
   route_info
   hotel_info
   transport_info
   event_info
   ↓
planner.generate_trip_plan()
   ↓
LLM 综合用户需求 + 工具结果生成 trip_data
   ↓
生成后工具执行：
   opening_hours_tool 检查开放时间
   image_tool 匹配地点图片
   ↓
storage.build_project_data()
   ↓
memory_manager 初始化记忆和版本历史
   ↓
add_version 保存初始版本
   ↓
命令行展示行程
   ↓
用户选择保存 / 修改 / 导出 / 查看记忆 / 恢复上一版
```


## 六、生成前工具与生成后工具

### 生成前工具

这些工具在大模型生成行程之前执行，会直接影响 `trip_data`：

- `map_tool`
- `weather_tool`
- `budget_tool`
- `poi_tool`
- `route_tool`
- `hotel_tool`
- `transport_tool`
- `event_tool`

### 生成后工具

这些工具依赖最终 `trip_data.plan` 中的具体地点，因此应在生成后执行：

- `image_tool`
- `opening_hours_tool`

## 九、进入 FastAPI 前的改造重点

- `trip_service.py` 里仍然有 `input()`
  例如 `edit_project()` 和 `load_project_flow()` 仍是 CLI 版写法，Web 接口不能直接复用。

- 建议拆分为 CLI 包装函数和纯服务函数
  例如：
  `edit_project_cli(project_data)`
  `edit_project_by_request(project_data, user_edit_request)`

- 项目读取要去掉“输入编号”依赖
  后续应拆出：
  `list_projects()`
  `load_project_by_filepath()`
  `load_project_by_id()`

- 导出返回值要更适合 Web
  后续接口建议返回：
  `success / export_type / filepath / download_url`

- 长耗时流程要准备 Web loading 状态
  当前生成流程会串联多个工具和一次 LLM，后续前端需要明确“生成中 / 导出中 / 失败重试”状态。


## 十、第 27 节建议改造方向

下一节进入 FastAPI 时，建议优先做下面几件事：

1. 保留 `trip_service.py` 作为服务层，不把业务逻辑塞进 FastAPI 路由。
2. 把 CLI 交互逻辑从服务逻辑中拆开。
3. 提供 `generate / edit / save / load / export / memory / restore` 这些 API。
4. 先做同步接口，跑通完整链路，再考虑异步任务和进度轮询。
